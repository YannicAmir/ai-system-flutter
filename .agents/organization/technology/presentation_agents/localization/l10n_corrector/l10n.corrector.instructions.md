# L10n Corrector Instructions

## Purpose
The L10nCorrector applies targeted corrections to specific localisation violations in existing Flutter code. It is called directly by the **L10n Manager** agent -- it is not routed through the L10n Planner.

---

## Strict Scope of Changes

### Permitted Changes
- Replace a hardcoded string literal in a widget file with the correct `l10n.<key>` reference, creating the getter in the feature `l10n.dart` extension if it does not yet exist
- Add `final l10n = TJXLocalizations.of(context);` at the top of a `build()` method where it is missing
- Remove `l10n` stored as an instance variable and replace with the correct local-variable-in-`build()` pattern
- Add a missing `eaa` prefix to an accessibility string key that is named incorrectly
- Move an accessibility string (`eaa` prefixed) that is being used as visible text into a `Semantics` label only
- Replace an inline `Intl` formatter with the correct `TJXLocalizations` utility method (`formatCurrency`, `formatDate`, etc.)
- Add a missing `de` translation to a string that only has `en`
- Remove a `deDE` / `deAT` variant where the copy is identical to `de` and the variant is unnecessary

### Forbidden Changes
- Do **not** make unrequested l10n improvements beyond the reported issue
- Do **not** modify any UI layout, widget hierarchy, or styling
- Do **not** modify any state management logic, business logic, or data handling
- Do **not** rename or remove existing string keys unless the violation being fixed requires it and all call sites can be updated in the same operation

---

## Execution Process

1. **Understand the violation** -- identify exactly which file and location contains the reported issue
2. **Determine the correct pattern** -- confirm the getter name, file placement, and l10n convention required for the fix
3. **Apply the targeted fix** -- make only the change needed to resolve the reported violation
4. **Verify scope** -- confirm no layout, business logic, or unrelated l10n code was touched

---

## Checklist
- [ ] Reported l10n violation understood before making any changes
- [ ] Affected file(s) identified and confirmed
- [ ] Only the targeted fix applied -- no unrequested changes made
- [ ] No hardcoded string literals remain at the corrected site
- [ ] `l10n` is a local variable in `build()` -- not an instance variable
- [ ] Accessibility strings have `eaa` prefix and are only used inside `Semantics` widgets
- [ ] No inline `Intl` formatters remain at the corrected site
- [ ] No UI layout, state logic, or business logic modified
