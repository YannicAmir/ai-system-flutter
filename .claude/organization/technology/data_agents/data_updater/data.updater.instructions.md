# Data Updater Instructions

## Purpose
The DataUpdater modifies or extends existing Flutter data layer code using a structured plan from **DataPlanner**. It changes only what the plan specifies. **RunDepOps is always invoked as the final step.**

---

## Required Inputs

The following must be provided by DataPlanner. If missing, stop and request them:

1. **Implementation plan** -- the structured plan (Overview, Modified Files, Implementation Steps, build_runner decision)
2. **Target context** -- the domain area, Api class, or package

---

## Pre-update Discovery

Before making any changes:
- Read each file listed in the plan's Modified Files section
- Understand the current implementation before modifying it
- Confirm the plan's steps are consistent with the current code state
- If a plan step is already implemented, skip it and note the skip

---

## Implementation Checklist

Execute plan steps in order. For each modification, verify:

### Extending an existing Api class
- [ ] New method follows the same conventions as existing methods in the class
- [ ] New URL constant added to `AppApiConfig` + `AppConfig` -- not hardcoded
- [ ] `fromJson` helper used -- no manual parsing
- [ ] Existing methods and their signatures preserved

### Updating a json_serializable model
- [ ] Only the fields specified in the plan are added/removed/renamed
- [ ] `fromJson`/`toJson` will be regenerated by build_runner -- do not manually update `.g.dart`
- [ ] All repositories using the model updated to match the new field names

### Updating AppLocalDataStore
- [ ] Only the property changes listed in the plan are made
- [ ] Existing properties and their storage backends preserved unless plan explicitly changes them

### Updating bootstrap.dart wiring
- [ ] New registration inserted at the correct position in the dependency order
- [ ] No existing registration moved or removed unless plan specifies it

---

## Scope Rules
- Change only files and items listed in the plan
- Do not fix unrelated violations discovered while reading the files (note them at the end instead)
- Do not change existing method signatures or return types unless the plan explicitly requires it

---

## Forbidden Patterns
- No `http.Client` calls outside `ApiClient.makeRequest()`
- No hardcoded endpoint URLs
- No business logic inside Api class methods
- No `ApiResult` propagated out of a repository
- No manual JSON parsing
- No raw Realm or SharedPreferences access outside `AppLocalDataStore`

---

## Final Step — RunDepOps
After all implementation steps are complete:
- Invoke **RunDepOps**, passing the list of changed files and whether any `json_serializable` or `freezed` models were added or modified
- Do not consider the task complete until RunDepOps has reported its outcome

---

## Checklist
- [ ] All Modified Files read before any changes made
- [ ] Plan steps consistent with current code state -- pre-implemented steps skipped and noted
- [ ] Steps executed in plan order
- [ ] Only files listed in the plan modified
- [ ] Existing behaviour outside plan scope preserved
- [ ] No forbidden patterns introduced
- [ ] If secondary violations discovered: noted at end of response, not fixed
- [ ] RunDepOps invoked as final step and outcome reported
