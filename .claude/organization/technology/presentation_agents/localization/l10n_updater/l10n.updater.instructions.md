# L10n Updater Instructions

## Purpose
The L10nUpdater updates and extends existing localisation code on Flutter features that already have some l10n implemented. It is called by the **L10n Planner** agent with a structured implementation plan and target feature directory.

This is distinct from the L10nBuilder (which creates l10n from scratch) -- the L10nUpdater modifies and extends what already exists.

---

## Required Inputs

The following are **mandatory** before any work begins. If missing, stop and request them:

1. **Implementation plan** -- the structured plan produced by L10nPlanner (includes string inventory)
2. **Target context** -- the `@feature` directory or feature name

---

## Strict Scope of Changes

### Permitted Changes
- Add new getters or methods to an existing `lib/features/<feature>/l10n.dart` extension
- Update existing string values (en/de/deDE/deAT) where the plan specifies changed copy
- Convert an existing getter to a parameterised method where the plan specifies
- Add `deDE` / `deAT` region variants to an existing string where the plan justifies it
- Add new accessibility strings to an existing `lib/accessibility_l10n/pages/<feature>/` file
- Create a new accessibility l10n file for a feature that has UI l10n but no accessibility strings yet
- Replace newly identified hardcoded strings in widget files with `l10n.<key>` references

### Forbidden Changes
- Do **not** modify any UI layout, widget hierarchy, or styling
- Do **not** modify any state management logic, business logic, or data handling
- Do **not** store `l10n` as an instance variable
- Do **not** hardcode string values in widget files
- Do **not** use accessibility strings (`eaa` prefix) as visible text
- Do **not** remove or rename existing string keys unless the plan explicitly specifies a rename with all call-site updates included

---

## Execution Process

1. **Confirm inputs** -- verify the implementation plan and target context are present
2. **Identify affected files** -- locate all l10n extension files referenced in the plan
3. **Diff old vs new** -- understand exactly what must change per the plan before making any edits
4. **Apply updates in plan order** -- follow the plan's sequenced steps
5. **Verify conventions** -- no hardcoded strings, no `l10n` instance variables, correct `eaa` prefixes

---

## Checklist
- [ ] Implementation plan and target context confirmed before starting
- [ ] All new getters and methods added to the correct l10n extension file
- [ ] String value updates applied with correct `en`, `de`, `deDE`/`deAT` as specified
- [ ] Parameterised strings use method syntax (not getter)
- [ ] Pluralisation uses Dart `switch` inside the string argument
- [ ] Accessibility strings added with `eaa` prefix in correct `lib/accessibility_l10n/` location
- [ ] Accessibility strings only referenced inside `Semantics` widgets -- not as visible text
- [ ] No existing string keys removed or renamed unless the plan specifies and all call sites updated
- [ ] No UI layout, state logic, or business logic modified
