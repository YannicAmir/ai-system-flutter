# L10n Builder Instructions

## Purpose
The L10nBuilder implements localisation for Flutter features that currently have no l10n code. It is called by the **L10n Planner** agent with a structured implementation plan and the target feature directory.

---

## Required Inputs

The following must be provided by L10nPlanner before work begins. If missing, stop and request them:

1. **Implementation plan** -- the structured plan produced by L10nPlanner (includes string inventory)
2. **Target feature directory** -- the `@feature` directory or path

---

## Strict Scope of Changes

### Permitted Actions
- Create `lib/features/<feature>/l10n.dart` with the extension on `TJXLocalizations` and all getters/methods from the plan
- Create `lib/accessibility_l10n/pages/<feature>/<feature>_l10n.dart` with accessibility string extension (using `eaa` prefix)
- Replace hardcoded string literals in widget `build()` methods with `l10n.<key>` references, adding `final l10n = TJXLocalizations.of(context);` at the top of `build()` where missing
- Replace inline `Intl` formatters with the appropriate `TJXLocalizations` utility method call

### Forbidden Actions
- Do **not** modify any UI layout, widget hierarchy, or styling
- Do **not** modify any state management logic, business logic, or data handling
- Do **not** store `l10n` as an instance variable -- it must always be a local variable in `build()`
- Do **not** hardcode string values in widget files -- all strings come from l10n extensions
- Do **not** use accessibility strings (`eaa` prefix) as visible text
- Do **not** add strings to `common_localizations.dart` -- the planner will specify if a string belongs there; only add to feature and accessibility files unless the plan explicitly states otherwise

---

## Execution Process

1. **Confirm inputs** -- verify the implementation plan and target feature directory are present
2. **Review the string inventory** -- understand every getter, method, and accessibility string before writing any code
3. **Create the feature l10n.dart first** -- extension must exist before it is referenced in widget files
4. **Create accessibility l10n files** -- if the plan includes accessibility strings
5. **Wire into widgets** -- add `final l10n = TJXLocalizations.of(context);` and replace hardcoded strings per the plan
6. **Verify conventions** -- no hardcoded strings remain, no `l10n` instance variables, correct `eaa` prefixes on accessibility strings

---

## Checklist
- [ ] Implementation plan and target feature directory confirmed before starting
- [ ] `lib/features/<feature>/l10n.dart` created with extension on `TJXLocalizations`
- [ ] All getters use getter syntax (no parameters); all parameterised strings use method syntax
- [ ] Pluralisation uses Dart `switch` inside the string argument
- [ ] `deDE` / `deAT` only included where the plan specifies and copy genuinely differs
- [ ] Accessibility strings created in `lib/accessibility_l10n/pages/<feature>/` with `eaa` prefix
- [ ] Accessibility strings only referenced inside `Semantics` widgets -- not as visible text
- [ ] `final l10n = TJXLocalizations.of(context);` added at the top of `build()` in all affected widgets
- [ ] No `l10n` stored as an instance variable anywhere
- [ ] No hardcoded string literals remain in widget files
- [ ] No inline `Intl` formatters -- `TJXLocalizations` utility methods used instead
- [ ] No UI layout, state logic, or business logic modified
